# Patrones de Diseño Aplicados - Módulo Contratación

## Resumen Ejecutivo

Este documento describe los **4 patrones de diseño obligatorios** implementados en el módulo de Contratación de Servicio, su justificación técnica y ubicación en el código.

---

## 1. Factory Method (Patrón Creacional)

### 📍 Ubicación
`apps/backend/src/modules/contratacion/domain/factories/orden-servicio.factory.ts`

### 🎯 Propósito
Encapsular la lógica de creación de instancias de `OrdenServicio` según el tipo de contratación (`POR_HORAS`, `POR_TAREA`, `POR_PAQUETE`) sin exponer los detalles de construcción al cliente.

### ✅ Problema que resuelve
- Evita la proliferación de condicionales `if-else` o `switch` en el código cliente
- Facilita agregar nuevos tipos de orden sin modificar código existente (Open/Closed Principle)
- Centraliza la lógica de inicialización y validación específica de cada tipo

### 💻 Implementación

**Productos concretos:**
- `OrdenPorHoras`: calcula costo base × horas + 10% margen
- `OrdenPorTarea`: aplica multiplicador según complejidad (BAJA: 1.0, MEDIA: 1.15, ALTA: 1.30)
- `OrdenPorPaquete`: calcula costo por entregable × cantidad - 10% descuento volumétrico

**Factory:**
```typescript
createOrden(tipo: TipoServicio, params: any): IOrdenServicio {
  switch (tipo) {
    case TipoServicio.POR_HORAS:
      return new OrdenPorHoras(params.tarifaPorHora, params.horasEstimadas);
    case TipoServicio.POR_TAREA:
      return new OrdenPorTarea(params.montoBase, params.complejidad);
    case TipoServicio.POR_PAQUETE:
      return new OrdenPorPaquete(params.montoPorEntregable, params.cantidadEntregables);
  }
}
```

## 2. Facade (Patrón Estructural)

### 📍 Ubicación
`apps/backend/src/modules/contratacion/infrastructure/facades/contratar-servicio.facade.ts`

### 🎯 Propósito
Simplificar la interacción con el subsistema complejo de contratación, proporcionando una interfaz unificada que orquesta múltiples operaciones.

### ✅ Problema que resuelve
- Reduce el acoplamiento entre el cliente (controlador) y múltiples servicios internos
- Oculta la complejidad de validación → cotización → creación de orden → auditoría
- Proporciona un punto de entrada único y claro para contratar un servicio

### 💻 Implementación

**Método principal:**
```typescript
async contratarServicio(datos: DatosContratacionDTO) {
  // PASO 1: Validar disponibilidad
  await this.validarDisponibilidad(solicitudId, clienteId);
  
  // PASO 2: Generar cotización
  const cotizacion = await this.generarCotizacion(datos);
  
  // PASO 3: Crear orden usando Factory
  const orden = await this.crearOrden(datos, cotizacion.id);
  
  // PASO 4: Actualizar solicitud
  await this.actualizarEstadoSolicitud(solicitudId, ACEPTADA);
  
  // PASO 5: Auditoría
  await this.registrarEnHistorial(orden.id, null, CREADA, 'Orden creada');
  
  return { orden, cotizacion };
}
```

**Subsistemas orquestados:**
- `PrismaService` (persistencia)
- `OrdenServicioFactory` (creación de órdenes)
- Lógica de validación de negocio
- Registro de auditoría

### 📊 Diagrama UML
```
┌───────────────────────┐
│   Controller          │
│   (Cliente)           │
└──────────┬────────────┘
           │
           │ contratarServicio()
           ▼
┌─────────────────────────────────┐
│ ContratarServicioFacade         │
│─────────────────────────────────│
│ + contratarServicio()           │
│ - validarDisponibilidad()       │

└──────┬────────┬────────┬────────┘

  ┌────────┐ ┌─────┐ ┌──────────┐
  │Prisma  │ │Fact.│ │Historial │
  │Service │ │     │ │Service   │
  └────────┘ └─────┘ └──────────┘
```

## 3. Command (Patrón de Comportamiento)

### 📍 Ubicación
`apps/backend/src/modules/contratacion/application/commands/cancelar-orden.command.ts`

### 🎯 Propósito
Encapsular acciones (cancelar, reprogramar, confirmar) como objetos reutilizables, permitiendo parametrizar operaciones, encolarlas, registrarlas y potencialmente deshacerlas.

### ✅ Problema que resuelve
- Desacopla el objeto que invoca la operación del que la ejecuta
- Facilita el registro de operaciones para auditoría
- Permite implementar lógica transaccional y rollback
- Centraliza validaciones específicas de cada acción

### 💻 Implementación

**Interfaz base:**
```typescript
interface ICommand {
  execute(): Promise<any>;
  validate(): Promise<boolean>;
  getDescription(): string;
}
```

**Comandos implementados:**

1. **CancelarOrdenCommand**
   - Valida que la orden no esté completada
   - Cambia estado a `CANCELADA`
   - Registra motivo y usuario que cancela
   - Audita en `HistorialEstado`

2. **ReprogramarOrdenCommand**
   - Valida que la nueva fecha sea futura
   - Actualiza `fechaInicio`
   - Cambia estado a `REPROGRAMADA`
   - Audita la acción

3. **ConfirmarEjecucionCommand**
   - Solo permite confirmar órdenes `EN_PROGRESO`
   - Cambia estado a `COMPLETADA`
   - Registra `fechaFinalizacion`
   - Audita con comentarios opcionales

### 📊 Diagrama UML
```
┌─────────────────────┐
│    ICommand         │
│─────────────────────│
│ + execute()         │
│ + validate()        │
│ + getDescription()  │
└──────────△──────────┘
           │
           │ implementa
           │
     ┌─────┼─────┬────────────┐
     │           │            │
┌────┴──────┐  ┌─┴──────┐  ┌─┴──────────┐
│Cancelar   │  │Reprog. │  │Confirmar   │
│Command    │  │Command │  │Command     │
└───────────┘  └────────┘  └────────────┘
```

**Uso en controlador:**
```typescript
@Patch(':id/cancelar')
async cancelarOrden(@Param('id') id: string, @Body() body) {
  return await this.cancelarCommand
    .setParams(id, body.motivo, body.usuarioId)
    .execute();
}
```

## 4. Template Method (Patrón de Comportamiento)

### 📍 Ubicación
`apps/backend/src/modules/contratacion/domain/services/proceso-contratacion.template.ts`

### 🎯 Propósito
Define el esqueleto del algoritmo de contratación en una operación base, delegando ciertos pasos a subclases sin cambiar la estructura del proceso.

### ✅ Problema que resuelve
- Garantiza que los pasos críticos se ejecuten siempre en el mismo orden
- Reutiliza código común entre diferentes tipos de contratación
- Permite personalizar pasos específicos sin duplicar lógica
- Implementa el principio Hollywood: "Don't call us, we'll call you"

### 💻 Implementación

**Flujo fijo (Template Method):**
```typescript
async ejecutarProcesoCompleto(datos: DatosProcesoContratacion) {
  // 1. Validación común (implementado en clase base)
  await this.validarDatosComunes(datos);
  
  // 2. Validación específica (delegado a subclase)
  await this.validarDatosEspecificos(datos);
  
  // 3. Calcular presupuesto (delegado a subclase)
  const presupuesto = await this.calcularPresupuesto(datos.parametrosServicio);
  
  // 4. Crear solicitud (implementado en clase base)
  const solicitudId = await this.crearSolicitud(datos, presupuesto);
  
  // 5. Generar cotización (implementado en clase base)
  const cotizacionId = await this.generarCotizacion(solicitudId, presupuesto);
  
  // 6. Aplicar ajustes específicos (delegado a subclase)
  const montoFinal = await this.aplicarAjustesEspecificos(presupuesto);
  
  // 7. Crear orden (implementado en clase base)
  const ordenId = await this.crearOrden(solicitudId, cotizacionId, datos, montoFinal);
  
  // 8. Notificación (hook opcional, puede sobrescribirse)
  await this.notificar(ordenId, datos.clienteId);
  
  return { solicitudId, cotizacionId, ordenId, montoFinal };
}
```

**Métodos abstractos (deben implementarse):**
- `validarDatosEspecificos()`: validación por tipo
- `calcularPresupuesto()`: cálculo específico
- `aplicarAjustesEspecificos()`: descuentos/márgenes
- `calcularDuracion()`: estimación de tiempo

**Implementaciones concretas:**
- `ContratacionPorHoras`
- `ContratacionPorTarea`
- `ContratacionPorPaquete`

### 📊 Diagrama UML
```
┌──────────────────────────────────────┐
│ ProcesoContratacionTemplate          │
│ (Clase Abstracta)                    │
├──────────────────────────────────────┤
│ + ejecutarProcesoCompleto() [FINAL]  │◄─── Template Method
│ # validarDatosComunes()              │
│ # crearSolicitud()                   │
│ # generarCotizacion()                │
│ # crearOrden()                       │
│ ─────────────────────────────────    │
│ + validarDatosEspecificos() [ABS]    │◄─── Métodos abstractos
│ + calcularPresupuesto() [ABS]        │
│ + aplicarAjustesEspecificos() [ABS]  │
│ + calcularDuracion() [ABS]           │
│ # notificar() [HOOK]                 │◄─── Hook opcional
└────────────────┬─────────────────────┘
                 │
        ┌────────┼────────┬──────────┐
        │                 │          │
┌───────▼────┐  ┌─────────▼───┐  ┌──▼─────────┐
│Contratación│  │Contratación │  │Contratación│
│PorHoras    │  │PorTarea     │  │PorPaquete  │
└────────────┘  └─────────────┘  └────────────┘
```

## Tabla Resumen

| Patrón | Tipo | Ubicación | Beneficio Principal |
|--------|------|-----------|---------------------|
| **Factory Method** | Creacional | `domain/factories/` | Encapsular creación de órdenes por tipo |
| **Facade** | Estructural | `infrastructure/facades/` | Simplificar proceso complejo de contratación |
| **Command** | Comportamiento | `application/commands/` | Encapsular acciones como objetos reutilizables |
| **Template Method** | Comportamiento | `domain/services/` | Definir flujo fijo con pasos personalizables |

---

## Justificación de la Selección

1. **Factory Method**: Necesario porque hay 3 tipos de orden con lógicas de cálculo diferentes.
2. **Facade**: El proceso de contratación involucra 5+ pasos con validaciones complejas.
3. **Command**: Las acciones (cancelar, reprogramar, confirmar) requieren auditoría y pueden implementarse como transacciones.
4. **Template Method**: El flujo de contratación tiene una estructura fija pero con variaciones por tipo de servicio.
